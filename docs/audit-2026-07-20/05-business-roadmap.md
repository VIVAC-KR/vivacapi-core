| 영역 | 파일 | 현재 상태 |

## 1. 성장/획득

| # | 항목 | 상태 | 난이도 | 의존성 |
|---|---|---|---|---|
| 1.1 | 재사용 가능한 리퍼럴 링크 | **✅ 완료**(branch `feature/reusable-referral-invite`, commit `2ffba07`) | 하 | 없음 |
| 1.2 | 스팟 상세 공개 공유 카드(OG 메타) | **⛔ 블로킹 — 전제 무효** | 하(프론트 범위) | 없음(하지만 전제 붕괴) |
| 1.3 | 공개 그룹 탐색/공유 진입점 | 제안 | (a)하/(b)중 | 없음 |
| 1.4 | 초대 현황 대시보드 | 제안 | 중 | 1.1 이후 권장 |

- **1.1 상세**: `crud/invite.py`의 `consume_invite_for_signup`에서 `group_uid is None`(일반 리퍼럴)인 경우 `status`를 `ACCEPTED`로 전환 안 하고 `PENDING` 유지 — 같은 링크로 반복 가입 가능해짐. 그룹초대는 기존과 동일 1회용 유지. 스키마/마이그레이션 변경 없음. 테스트 1건 추가, 전체 267개 통과(무관한 `test_cors.py` 6개 기존 실패 제외). **어뷰징 방지는 이번 스코프에서 의도적 제외**(후속 고려사항).
  - ⚠️ **이 결정은 `docs/projects/spot-invites.md`(2026-07-16)의 원래 설계("1회용")를 뒤집는다.** 상세는 [08-cross-cutting-issues.md](./08-cross-cutting-issues.md) 참고.
- **1.2 상세**: 재조사 결과 VIVAC-frontend에 스팟 상세 페이지(`/spots/{uid}`) 자체가 없음 — 커밋 `788291c`("NextAuth v5 전환 및 홈 개편", 2026-07-13)에서 `features/spots/*` 전체 제거. 백엔드(`GET /v1/explore/spots/{uid}`)는 정상이지만 붙일 화면이 없음. OG 메타만 추가하는 원 스코프로 진행 불가 — 스팟 상세 페이지 자체를 새로 설계해야 함. **사용자 판단 필요, 재개 안 함.**
- **1.3 상세**: 애초 "PUBLIC 그룹도 로그인 필요"로 짚었으나, 코드 확인 결과 그룹 상세/spots는 이미 비로그인 열람 가능(`spot_groups.py:52-66,153-155,193-195`). 실제 갭은 "발견 가능성" — (a) 공유 URL 노출+공유버튼(백엔드 변경 없음), (b) `GET /v1/groups/public` 디스커버리 엔드포인트(`crud/spot_group.py`에 유사 쿼리패턴 존재, 재사용 가능)
- **1.4 상세**: 데이터(`referred_by_uid`, `Invite.accepted_by_uid`)는 이미 쌓이는데 조회 API가 없는 전형적 "데이터는 있고 피드백 루프만 없는" 케이스

## 2. 리텐션/인게이지먼트

| # | 항목 | 난이도 | 의존성 |
|---|---|---|---|
| 2.1 | 개인 찜(즐겨찾기) | 하 | 2.3의 선행기능 |
| 2.2 | 캠핑 다이어리(내 리뷰 타임라인) | 하 | 없음 |
| 2.3 | 찜한 스팟 상태변경 알림 | **상** | 2.1, 4.1·4.2와 강결합 — 가장 늦게 착수 권장 |
| 2.4 | 그룹 활동 피드 | 중 | 없음 |

- 2.1: 그룹기능은 무거운 흐름(생성·초대·멤버관리) — 원클릭 개인저장 니즈 미충족. `spot_groups`/`spot_group_spots` 재사용 또는 별도 `favorites` 조인테이블 중 택1
- 2.3: 트리거 필드(`pipeline_status`, `trust_tier`)는 있으나 **알림 발송 인프라(FCM/APNs, 구독테이블) 자체가 전무** — 신규 인프라 구축 필요

## 3. 수익화/파트너십

| # | 항목 | 난이도 | 의존성 |
|---|---|---|---|
| 3.1 | 예약 클릭 어필리에이트 | 하 | 3.4와 이벤트로깅 공유 |
| 3.2 | 사업자 인증 파트너십(Verified Operator) | 중 | 4.4 이후 권장 |
| 3.3 | 피처드 리스팅(Sponsored Placement) | 중 | 3.2 이후 |
| 3.4 | 문의 리드 판매(Pay-per-Lead) | 하 | 3.1과 인프라 공유 |

- 3.1: `booking_url` 필드는 이미 있음, 클릭트래킹 인프라만 얹으면 됨 — 결제 인프라 없이 시작 가능
- 3.2: `trust_tier`/`pipeline_status` 검증체계 위에 "본인확인 사업자" 신뢰레이어 얹는 구조, 기존 MANAGER 권한 패턴 재사용. 셀프관리 권한을 월구독 유료티어로 판매 구상
- 3.3: `is_sponsored` 플래그+정렬로직만으로 시작, 초기엔 결제자동화 없이 수동 인보이스

## 4. 신뢰/데이터품질

| # | 항목 | 상태 | 난이도 | 의존성 |
|---|---|---|---|---|
| 4.1 | 스팟 폐쇄/접근불가 신고 | 제안 | 중 | **4.4를 먼저 처리 권장** |
| 4.2 | trust_tier 신선도 기반 자동 감쇠 | **✅ 완료 — PR #117**(branch `feature/trust-tier-freshness`, commit `b0fe782`) | 중 | 없음, 4.1과 병행 가능 |
| 4.3 | 리뷰 "지금도 유효한가요?" 마이크로시그널 | 제안 | 하 | 없음(단독 가치, 4.1과 결합 시 시너지) |
| 4.4 | 검증 담당자 재할당 API | **✅ 완료 — PR #118**(branch `feature/spot-assignment-reassign`, commits `bf0d6a9`, `bfaefac`) | 하 | 없음 — **4.1 착수 전 먼저 처리 권장** |

- 4.1: 노지 스팟이 사유지화·산불통제 등으로 조용히 폐쇄되는 경우가 많은데, 스팟 자체 신고 모델이 없음(`SpotReport`류 0건) — "잘못된 위치정보로 헛걸음"이 가장 치명적 신뢰훼손 시나리오
- 4.2(제안 당시 문제의식, 완료됨): `trust_tier`(1~3)는 갱신 트리거 없음 — 검증시점 컬럼 자체가 없음(`identity_verified_at`은 User 전용). `Spot.last_verified_at` 신설+배치job으로 자동하향 제안 → **PR #117로 구현 완료**
- 4.4(제안 당시 문제의식, 완료됨): `assigned_to_uid`가 `SpotEditableFields`에 없어 PATCH 불가, `POST /internal/spots/assignments`도 NULL인 spot만 동작 — 재할당/해제 수단 자체가 없었음. 담당자 휴가·과부하 시(특히 4.1 신고급증) 파이프라인 막힘 우려 → **PR #118로 재할당/해제 API 구현 완료**
